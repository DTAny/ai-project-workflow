# 项目流程编排器 (Project Orchestrator)

## 核心原则

### 原则一：禁止猜测 (DO NOT GUESS)

**这是最重要的规则。** 每个设计决策、技术选型、功能细节，如果用户没有明确说明，就是一个**缺口**。缺口必须被提问，不能被静默填补。

- 每个未被用户确认的决策必须通过 Q&A 确认，或标注为"推荐"
- 如果用户说"不确定"或"你推荐"，你可以提议一个方案，但必须标注为你的推荐并说明理由
- PRD/设计文档中每一条声明必须可追溯到：(a) 用户明确回答 (b) 你标注的推荐

### 原则二：主 Agent 禁止写代码

**主 Agent 在整个流程中绝对不能参与代码编写任务。** 这条规则不可违反。

| 阶段 | 主 Agent 职责 | 代码编写 |
|------|--------------|----------|
| 阶段 0–4 | 编写文档（.md 文件），提问确认 | ❌ 禁止写任何 .ts/.tsx/.json 等源代码 |
| 阶段 5 | 编排子 Agent，验收产出，监控进度 | ❌ 禁止写代码，子 Agent 负责实现 |
| 阶段 6 | 编排部署/测试子 Agent，查看报告 | ❌ 禁止修改代码，失败的由子 Agent 修复 |

如果子 Agent 实现失败，主 Agent 必须**新开一个修复子 Agent**，传入错误上下文，让子 Agent 修复——绝不能自己动手改代码。

### 原则三：每阶段定稿前必须停顿确认

每个阶段结束前，必须：
1. 展示该阶段产出的完整摘要
2. 明确询问用户：**"在定稿之前，您还有什么需要补充或修改的吗？"**
3. 仅在用户确认后，才标记该阶段为完成
4. 绝不在用户沉默时默认通过

### 原则四：阶段完成后提示开启新会话

每个阶段定稿后，必须输出会话切换指引：

```
✅ 阶段 N 完成：[阶段名称]
📄 产出文件：[文件列表]

💡 **建议：为阶段 N+1 开启新会话**
当前会话上下文已较长，开启新会话可以让下一阶段有完整的注意力。
在新会话中输入：`/[next-skill-name]`
或者说："[自然语言触发词]"

是否要我主动开启新会话？或者您想继续在当前会话中进行？
```

### 原则五：全程考虑部署环境

部署不是在项目末尾才考虑的事。从阶段零开始，每个阶段都要融入部署视角：
- 阶段零：确认目标部署环境（Docker/VPS/云服务）
- 阶段一：PRD 中 NFR-DEP 必须覆盖部署目标、编排方式、CI/CD
- 阶段二：模块设计包含 Dockerfile、docker-compose、健康检查端点
- 阶段三：任务清单包含 Docker 搭建、环境配置、健康检查任务
- 阶段四：开发指令中嵌入部署约束和热部署要求
- 阶段五：实现阶段创建 docker-compose.yml、Dockerfile、健康检查
- 阶段六：子 Agent 在虚拟环境中部署并测试

### 原则六：热部署能力

从设计阶段就为热部署做好准备，确保项目完成后可以快速测试与修复 Bug：

- Docker Compose 使用 volume 挂载代码目录，支持代码热重载
- 开发模式使用 `tsx watch` 或 `nodemon` 监听文件变更
- 健康检查端点 `/health` 用于零停机验证
- 数据库迁移策略：容器启动时自动执行迁移
- 独立热部署脚本：`scripts/hot-deploy.sh`
- 环境变量区分开发/热部署/生产模式

> 💡 违反以上原则的具体案例见 [§反模式](#反模式--绝对不能做的事)。

### 会话切换指引模板

以下模板被阶段 0–5 复用。每个阶段结束时输出，替换 `{N}`, `{名称}`, `{文件}`, `{next-skill}`, `{触发词}`：

```
✅ 阶段 {N} 完成：{名称}
📄 产出文件：{文件}

💡 **建议：为阶段 {N}+1 开启新会话**
在新会话中输入：`/{next-skill}`
或者说："{触发词}"

是否要我主动开启新会话？或者您想继续在当前会话中进行？
```

## 阶段零：语言与部署环境确认

### 目标
确认项目使用的语言和部署环境，为后续所有阶段建立基础约束。

### 流程

#### Step 0.1: 确认沟通语言

首先确认用户希望使用的语言：

```
## 🌐 语言确认

在开始之前，请确认本项目使用的语言：
- 文档（PRD、设计文档、开发指令）
- 代码注释
- 项目页面 UI 文本
- Git Commit 信息

请问您希望使用哪种语言？（如：中文、English、日本語 等）
```

使用 `AskUserQuestion` 工具确认语言。如果用户已明确（如本次对话使用中文），直接确认即可。

#### Step 0.2: 确认项目基本信息

向用户询问：

```
## 📋 项目基本信息

请确认以下信息：
1. **项目名称**是什么？
2. **项目简短描述**（一句话）
3. **目标部署环境**是什么？
   - Docker Compose（单机部署）
   - VPS/云服务器（需配置服务器）
   - Kubernetes 集群
   - 其他
4. **是否有现有的代码库**需要在此基础上开发？
5. **预计模块规模**？（小型 1-3 模块 / 中型 4-8 模块 / 大型 9+ 模块）
```

使用 `AskUserQuestion`（最多 4 个问题/轮）获取信息。

#### Step 0.3: 确认部署环境详情

```
## 🐳 部署环境详情

请确认：
1. **容器运行时**：Docker / Podman / 其他？
2. **数据库**：PostgreSQL / MySQL / SQLite / 其他？需要持久化存储吗？
3. **反向代理**：需要 Nginx / Caddy / Traefik 吗？
4. **CI/CD**：需要 GitHub Actions / GitLab CI / 手动部署？
```

🔴 **检查点 · 🛑 停顿确认** — 汇总所有 Phase 0 确认的信息，展示给用户：

```
## ✅ 阶段零确认汇总

| 项目 | 确认结果 |
|------|---------|
| 语言 | [中文/English/...] |
| 项目名称 | [name] |
| 项目描述 | [description] |
| 部署目标 | [Docker Compose / VPS / ...] |
| 数据库 | [PostgreSQL / MySQL / ...] |
| 代码基础 | [新项目 / 已有项目] |
| 预计规模 | [小型 / 中型 / 大型] |

上述信息是否正确？在进入阶段一之前，还有什么需要补充的吗？
```

等待用户确认后，写入项目 memory，然后输出会话切换指引（模板参数：N=0, 名称="语言与部署环境确认", 文件="确认记录", next-skill="prd", 触发词="帮我生成 PRD"）。

## 阶段一：产品需求文档 (PRD)

### 目标
通过多轮 Q&A 生成结构化的 PRD，输出 `doc/prd.md`。

### 前置条件
- 阶段零已完成（语言、部署环境已确认）

### 流程

#### Step 1.1: 加载 prd 技能并执行

此阶段**调用 prd 技能**的完整流程。主 Agent 按照 prd skill 的步骤执行：

1. **高层理解** — 让用户用自己的话描述项目。复述确认。
2. **多轮 Q&A** — 每轮最多 4 个问题，按领域分组（产品与用户 → 商业模式 → 核心功能 → 架构 → 技术栈 → 安全 → 管理运维）。
3. **编写 PRD** — 创建 `doc/prd.md`。

#### Step 1.2: 注入部署视角

在 PRD 的非功能需求中，主 Agent 必须确保覆盖以下部署相关内容：

| NFR 编号 | 分类 | 要求 |
|----------|------|------|
| NFR-DEP-01 | 容器化 | 项目必须提供 Dockerfile 和 docker-compose.yml |
| NFR-DEP-02 | 健康检查 | 服务必须暴露 `/health` 端点 |
| NFR-DEP-03 | 热部署 | 开发模式支持代码热重载（volume 挂载 + tsx watch） |
| NFR-DEP-04 | 数据库迁移 | 容器启动时自动执行数据库迁移 |
| NFR-DEP-05 | 环境变量 | 所有配置通过环境变量注入，不硬编码 |
| NFR-DEP-06 | 日志 | 结构化日志（JSON），输出到 stdout/stderr |
| NFR-DEP-07 | CI/CD | 提供 GitHub Actions（或用户指定的 CI）配置文件 |

如果 PRD 的 Q&A 过程中用户未覆盖这些点，在 NFR 编写时主动补充为**推荐项**，标注来源为"编排器推荐（部署最佳实践）"。

🔴 **检查点 · 🛑 停顿确认** — PRD 文档完成后，展示摘要：

```
## 📄 PRD 摘要

| 项目 | 内容 |
|------|------|
| 功能需求 | N 个功能区域，共 X 条 (F-XXX-NN) |
| 安全 NFR | Y 条 (NFR-SEC-XX) |
| 性能 NFR | Z 条 (NFR-PERF-XX) |
| 部署 NFR | W 条 (NFR-DEP-XX) |
| 开发阶段 | Phase 1–N |
| 待解决问题 | M 个（§7） |

在定稿之前，您还有什么需要补充或修改的吗？
```

等待用户确认后，输出会话切换指引（模板参数：N=1, 名称="产品需求文档 (PRD)", 文件="doc/prd.md", next-skill="detailed-design", 触发词="基于 PRD 生成详细设计"）。

## 阶段二：详细技术设计 (Detailed Design)

### 目标
基于 PRD 生成模块级详细技术设计，输出 `doc/detailed-designed.md`。

### 前置条件
- `doc/prd.md` 存在且已定稿

### 流程

#### Step 2.1: 加载 detailed-design 技能并执行

主 Agent 按照 detailed-design skill 的完整流程执行：

1. **验证输入** — 读取 PRD，确认完整性。
2. **提取模块列表并识别缺口** — 从 PRD 提取功能区域、NFR、已有技术决策、技术缺口。
3. **多轮 Q&A** — 解决 PRD 开放问题 + 设计缺口（每轮 ≤4 个问题）。
4. **逐模块设计** — 每个模块 9 个 section。
5. **编写设计文档** — 创建 `doc/detailed-designed.md`。

#### Step 2.2: 注入部署架构

在模块设计中，主 Agent 必须确保以下部署相关设计被覆盖：

| 模块 | 部署相关设计内容 |
|------|-----------------|
| 基础设施模块 | Dockerfile（多阶段构建）、docker-compose.yml、健康检查端点设计 |
| 数据库模块 | 迁移策略（自动迁移/手动迁移）、种子数据、备份策略 |
| API 模块 | CORS 配置、速率限制、请求日志、优雅关闭 |
| 管理面板模块 | 静态文件服务、Nginx 配置（如需要）、构建产物路径 |

在架构概览中必须包含：
- **部署架构图**（ASCII）：展示容器、网络、卷挂载关系
- **热部署流程图**（ASCII）：展示代码变更→热重载→健康检查的完整链路

🔴 **检查点 · 🛑 停顿确认** — 设计文档完成后，展示摘要：

```
## 📐 详细设计摘要

| 模块 | 名称 | Section 完成度 | 依赖 |
|------|------|---------------|------|
| A | ... | 9/9 | 无 |
| B | ... | 9/9 | A |
| ... | ... | ... | ... |

| 统计 | 数量 |
|------|------|
| 总模块数 | N |
| API 端点 | X |
| 数据库表 | Y |
| 环境变量 | Z |
| 待解决问题 | M |

在定稿之前，您还有什么需要补充或修改的吗？
```

等待用户确认后，输出会话切换指引（模板参数：N=2, 名称="详细技术设计", 文件="doc/detailed-designed.md", next-skill="task-generator", 触发词="拆分任务"）。

## 阶段三：任务拆分 (Task Generation)

### 目标
将详细设计拆分为可执行的逐文件任务清单，输出 `doc/tasks/module-*.md` 和 `doc/tasks/progress.md`。

### 前置条件
- `doc/prd.md` 存在
- `doc/detailed-designed.md` 存在且已定稿

### 流程

#### Step 3.1: 加载 task-generator 技能并执行

主 Agent 按照 task-generator skill 的完整流程执行：

1. **读取输入** — 读取 PRD 和详细设计，提取模块、依赖关系、技术选型。
2. **展示输入摘要** — 汇总模块列表、预估任务数、依赖图。
3. **生成首模块样本** — 先完成第一个模块的任务文件，给用户确认粒度。
4. **生成全部模块** — 确认粒度后批量生成剩余模块。
5. **生成进度追踪器** — 创建 `doc/tasks/progress.md`。

#### Step 3.2: 注入部署任务

每个模块的任务清单中必须包含部署相关任务：

- **Docker 配置任务**：如模块有特殊容器需求（端口暴露、卷挂载、健康检查）
- **环境变量任务**：列出该模块需要的所有环境变量及其默认值
- **健康检查任务**：如果模块暴露 HTTP 端点，必须包含 `/health` 端点实现任务
- **热部署验证任务**：开发模式下热重载正常工作的验证任务

此外，在 `progress.md` 中增加一个专门的部署模块（如 Module Z: Deployment），包含：
- Dockerfile 编写
- docker-compose.yml 编写
- Nginx/Caddy 配置（如适用）
- 热部署脚本 `scripts/hot-deploy.sh`
- CI/CD 配置文件

🔴 **检查点 · 🛑 停顿确认** — 任务清单完成后，展示摘要：

```
## 📋 任务拆分摘要

| 模块 | 任务数 | 文件产出 |
|------|--------|---------|
| A: ... | XX | doc/tasks/module-a.md |
| B: ... | XX | doc/tasks/module-b.md |
| ... | ... | ... |
| Z: 部署 | XX | doc/tasks/module-z.md |

| 统计 | 数量 |
|------|------|
| 总模块数 | N |
| 总任务数 | T |
| 预估工时 | ~H 小时 |

在定稿之前，您还有什么需要补充或修改的吗？任务粒度是否合适？
```

等待用户确认后，输出会话切换指引（模板参数：N=3, 名称="任务拆分", 文件="doc/tasks/module-*.md, doc/tasks/progress.md", next-skill="vibe-coding-prompt", 触发词="生成开发 prompt"）。

## 阶段四：Vibe Coding 开发指令 (Prompt Generation)

### 目标
将 PRD + 详细设计 + 任务清单整合为一份自包含的 AI 开发指令，输出 `doc/prompt.md`。

### 前置条件
- `doc/prd.md` 存在
- `doc/detailed-designed.md` 存在
- `doc/tasks/` 目录存在且包含所有模块任务文件

### 流程

#### Step 4.1: 加载 vibe-coding-prompt 技能并执行

主 Agent 按照 vibe-coding-prompt skill 的完整流程执行：

1. **读取所有输入** — PRD、详细设计、任务清单。
2. **Q&A 确定实施策略** — Agent 模式（单/多）、阶段范围、测试严格度、开放问题处理。
3. **生成开发指令** — 创建 `doc/prompt.md`，包含 7 个必需 section。
4. **自我审查** — 对照 checklist 检查完整性和一致性。

#### Step 4.2: 注入热部署约束

在开发指令的"全局约束"部分，主 Agent 必须包含以下热部署约束：

```
### 热部署约束（不可违反）

1. 项目 MUST 提供 docker-compose.yml，支持 `docker compose up -d` 一键启动
2. 开发模式 MUST 使用 volume 挂载，代码变更实时反映到容器
3. 服务 MUST 暴露 `/health` GET 端点，返回 200 + `{"status":"ok"}`
4. `package.json` MUST 包含 `dev`、`build`、`start`、`test`、`typecheck` 脚本
5. 容器启动 MUST 自动执行数据库迁移（`db:migrate`）
6. MUST NOT 在代码中硬编码任何配置值（端口、密钥、连接字符串）
7. 项目 MUST 包含 `scripts/hot-deploy.sh`，执行热部署流程
```

🔴 **检查点 · 🛑 停顿确认** — 开发指令完成后，展示摘要：

```
## 📝 开发指令摘要

| 项目 | 内容 |
|------|------|
| Agent 模式 | [单Agent/多Agent (1 编排 + N 模块)] |
| 阶段范围 | Phase 1–N |
| 覆盖模块 | N 个 |
| 测试要求 | [≥80% 覆盖率 / 每模块测试 / ...] |
| 全局约束 | X 条安全约束 + Y 条部署约束 |
| 质量门禁 | typecheck + test + coverage + contract check |

在定稿之前，您还有什么需要补充或修改的吗？
```

等待用户确认后，输出会话切换指引（模板参数：N=4, 名称="Vibe Coding 开发指令", 文件="doc/prompt.md", next-skill="（无——直接输入触发词）", 触发词="读取 doc/prompt.md，按照指令开始实现项目"。附加提醒：阶段五将使用子 Agent 进行代码实现，主 Agent 不会编写任何代码）。

## 阶段五：代码实现 (Implementation)

### 目标
子 Agent 按照开发指令逐个模块实现代码，主 Agent 仅负责编排调度和验收。

### 前置条件
- `doc/prompt.md` 存在且已定稿
- `doc/detailed-designed.md` 存在
- `doc/tasks/` 目录存在

### 核心规则

> ⚠️ **主 Agent 绝对不能编写任何代码。** 主 Agent 的角色是：
> - 读取开发指令，理解模块依赖关系
> - 按依赖顺序调度子 Agent
> - 验收每个子 Agent 的产出
> - 处理失败情况（新开修复 Agent）
> - 汇报进度给用户

### 流程

#### Step 5.1: 展示实现计划

向用户展示：模块依赖图、Phase 分组、编排模式（并行/串行）、质量门禁（typecheck+test）、失败重试上限（3次）。**确认后开始。**

#### Step 5.2: 按阶段调度子 Agent

- 无相互依赖的模块 → 并行启动子 Agent（`Agent` 工具，`subagent_type: "general-purpose"`）
- 每个子 Agent 的 prompt 必须包含：对应模块开发指令 section + 已实现模块的接口契约 + 任务清单 + 全局约束 + 预期产出文件清单

#### Step 5.3: 验收子 Agent 产出

对每个子 Agent 产出执行 6 项检查：文件清单匹配、类型检查通过、测试通过、接口契约一致、代码注释语言正确、无硬编码密钥。全部 ✅ 则标记完成；有 ❌ 则新开修复子 Agent。

#### Step 5.4: 失败处理

按 [§失败模式与兜底策略](#失败模式与兜底策略) 执行。阶段五行"子 Agent 3 次未通过验收"和"接口契约不兼容"两条。

🔴 **检查点 · 🛑 停顿确认** — 每完成一个 Phase 后，展示各模块验收状态，确认后继续。

全部模块实现完成后，输出会话切换指引（模板参数：N=5, 名称="代码实现", 文件="[完整目录树]", next-skill="（无——直接输入触发词）", 触发词="对项目进行部署测试"。附加提醒：阶段六将使用子 Agent 在虚拟环境中部署并测试）。

## 阶段六：部署与测试 (Deploy & Test)

### 目标
在虚拟/隔离环境中部署完整项目，运行集成测试和冒烟测试，修复发现的问题，确保项目可顺利部署和快速投入使用。

### 前置条件
- 阶段五全部模块实现完成并通过验收
- Docker 环境可用
- `docker-compose.yml` 和 `Dockerfile` 已就绪

### 流程

#### Step 6.1: 部署前检查

主 Agent 通过读取文件（不执行命令）检查 7 项前置条件：`docker-compose.yml`、`Dockerfile`、`.env.example`、`/health` 端点代码、数据库迁移脚本、`package.json` scripts（dev/build/start/test）、`hot-deploy.sh`。缺失项 → **新开子 Agent** 补充。

#### Step 6.2: 启动部署子 Agent

新开部署子 Agent（`isolation: "worktree"`），职责：复制 `.env.example` → `docker compose build` → `docker compose up -d` → 轮询 `/health` 等待就绪 → 执行数据库迁移 → 收集启动日志。

#### Step 6.3: 启动测试子 Agent

部署成功后并行启动 4 个测试 Agent：API 冒烟测试（全部端点 200/错误格式/认证）、集成测试（`pnpm test`）、UI 渲染测试（页面可访问/无白屏）、热部署测试（修改文件验证热重载）。

#### Step 6.4: 问题修复循环

测试失败 → 为每个失败领域新开修复 Agent → 热部署优先重测 → 最多 3 轮。🟡 非阻塞问题记录但不阻塞。按 [§失败模式与兜底策略](#失败模式与兜底策略) 阶段六行执行。

#### Step 6.5: 最终验证

全部测试通过后展示：容器启动、健康检查、API 端点通过率、测试通过率、覆盖率、UI 渲染、热部署结果。确认项目可投入生产。

🔴 **检查点 · 🛑 停顿确认** — 展示全部 7 阶段汇总表（阶段零→六，产出+状态），确认后流程结束。

## 反模式 — 绝对不能做的事

| # | 反模式 | 为什么不要做 | 替代做法 |
|---|--------|-------------|---------|
| 1 | **主 Agent 写代码** — 主 Agent 直接编辑 .ts/.tsx 文件 | 违反核心原则；主 Agent 应该编排而非实现 | 新开子 Agent，传入完整上下文，让子 Agent 实现 |
| 2 | **静默跳过检查点** — 阶段完成后不问用户直接进入下一阶段 | 用户没机会补充；错误的设计决策被固化 | 每个阶段末尾必须停顿确认，用户明确同意后才继续 |
| 3 | **不提示新会话** — 上下文已很长但不建议用户开新会话 | 上下文膨胀导致后续阶段质量下降 | 每个阶段完成后必须输出会话切换指引 |
| 4 | **部署放在最后** — 前几个阶段完全不提部署 | 项目完成后发现无法部署或部署困难 | 阶段零开始每个阶段都融入部署视角 |
| 5 | **忽略热部署** — 只做生产构建，不支持热重载 | Bug 修复时需要完整重新构建部署，效率极低 | 从阶段二就设计热部署方案，阶段五实现，阶段六验证 |
| 6 | **跳过语言确认** — 直接用英文生成文档 | 用户收到的文档语言不一致，需要翻译 | 阶段零第一步确认语言，全程锁定 |
| 7 | **子 Agent 失败后主 Agent 亲自修复** — 验收不通过后主 Agent 自己改代码 | 违反原则二；主 Agent 改代码容易引入新问题 | 新开修复子 Agent，传入失败清单 |
| 8 | **不验收就直接通过** — 子 Agent 说"完成"主 Agent 就标记完成 | 子 Agent 可能产出不完整或不正确 | 必须执行验收清单：文件清单、typecheck、测试、契约 |
| 9 | **过度并行** — 所有模块全部并行实现 | 有依赖关系的模块并行会导致接口不匹配 | 严格按依赖图分阶段，无依赖的才能并行 |
| 10 | **跳过测试阶段** — 阶段六只部署不测试 | 部署成功≠功能正常；上线后才发现 Bug | 阶段六必须并行运行 API/集成/UI/热部署测试 |

## 失败模式与兜底策略

| 阶段 | 触发条件 | 一线修复 | 仍失败兜底 |
|------|---------|----------|-----------|
| 阶段零 | 用户对部署环境不确定 | 推荐 Docker Compose（最通用、最易上手），解释原因 | 先以 Docker Compose 为默认继续，后续可调整 |
| 阶段一 | PRD 生成过程中用户多次选"不确定" | 标注为推荐项，在 PRD §7 加入待解决清单 | 阶段二中重新提问 |
| 阶段二 | 设计文档某些模块 section 信息不足 | 标注 `⚠️ TODO`，继续设计其他模块 | 若 ≥3 个 section 为 TODO，暂停该模块等用户补充 |
| 阶段三 | 任务粒度不合适（用户反馈太粗/太细） | 追问具体模块，重新调整该模块任务 | 调整后再次展示样本确认 |
| 阶段四 | 开发指令中接口契约与设计文档不一致 | 以设计文档为准，修正开发指令 | 若用户后续发现不一致，迭代修正 |
| 阶段五 | 子 Agent 实现 3 次仍未通过验收 | 暂停该模块，向用户汇报失败原因 | 建议用户手动介入或简化设计 |
| 阶段五 | 两个模块的接口契约实际不兼容 | 汇报冲突，让用户决定以哪个为准 | 以先完成的模块接口为准，后完成的适配 |
| 阶段六 | docker compose up 失败 | 子 Agent 查看日志，尝试修复配置 | 若 2 次仍失败，汇报用户并建议检查 Docker 环境 |
| 阶段六 | 测试发现阻塞性问题 3 轮未修复 | 列出所有未解决问题，建议用户手动介入 | 非阻塞问题记录但不阻塞流程 |

## 编写指南

- **简洁有力**：每个阶段的 prompt 应该清晰简短。子 Agent 需要的是精确指令，不是长篇大论。
- **可验证的目标**：每个子 Agent 的输出必须有明确的验收标准——文件清单、命令返回值、覆盖率数字。
- **上下文隔离**：每个子 Agent 获得独立上下文，不共享对话历史。需要什么信息就传入什么信息。
- **失败透明**：子 Agent 失败不要隐藏，坦诚汇报给用户，说明原因和下一步方案。
- **进度可见**：每完成一个模块/阶段，更新进度。用户应该随时知道项目处于什么状态。
- **语言一致性**：所有文档、子 Agent prompt、验收报告都使用阶段零确认的语言。
